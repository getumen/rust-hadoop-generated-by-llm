# ADR: io_uring を ChunkServer に採用しない

**日付:** 2026-06-03  
**ステータス:** 採用拒否 (Rejected)

---

## 背景

ChunkServer の `write_block`・`read_block` RPC が `std::fs` をブロッキングで呼び出していた。
io_uring（Linux カーネル 5.6+ の非同期 I/O インターフェース）に切り替えることで、
syscall 削減・レイテンシ改善が期待できると考え、2 段階で実装・評価した。

---

## 実装した 2 アーキテクチャ

### Phase 1: `spawn_blocking + tokio_uring::start()` per RPC

RPC ごとに `tokio_uring::start()` を呼び出す最もシンプルな実装。

**結果（Docker Linux ARM64）:**

| 操作 | std::fs | io_uring | 差 |
|------|---------|----------|----|
| read 4KB | 1.8 µs | 60.7 µs | **34x 遅い** |
| read 64KB | 5.0 µs | 63.9 µs | **13x 遅い** |
| write 4KB | 385 µs | 508 µs | -24% |

**原因:** `tokio_uring::start()` は毎回 io_uring リングを作成・登録する（~60µs）。
この初期化コストが I/O 時間を完全に上回る。

### Phase 2: 永続 IoUringPool（起動時にリングを1度だけ作成）

専用スレッドが `tokio_uring::start()` を一度だけ呼び起動し続ける。
RPC は `tokio::sync::mpsc` チャネルで I/O リクエストを送り、
`tokio::sync::oneshot` で結果を受け取る。

**結果（Docker Linux ARM64）:**

| 操作 | std::fs | io_uring pool | 差 |
|------|---------|---------------|----|
| read 4KB | 1.8 µs | 68.7 µs | **38x 遅い** |
| read 64KB | 4.6 µs | 70.9 µs | **15x 遅い** |
| read 1MB | 45 µs | 141 µs | **3x 遅い** |
| write 4KB | 388 µs | 579 µs | -33% |
| write 64KB | 421 µs | 484 µs | -13% |
| write 1MB | 2.06 ms | 1.73 ms | +16% |

**原因:** リング初期化コストは解消されたが、
`mpsc + oneshot` チャネルのラウンドトリップが約 65〜70µs かかる。
小〜中ブロックの read はページキャッシュから ~2µs で返るため、
チャネル経由で別スレッドに委譲するコストが圧倒的に大きい。

---

## 判断

**io_uring を採用しない。**

### 理由

1. **シングル op レイテンシが悪化する**: 全てのブロックサイズで read が遅くなる。
   LRU キャッシュがミスした場合も io_uring 経由では遅い。

2. **チャネルオーバーヘッドは構造的問題**: `tokio-uring` は専用ランタイムを必要とし、
   標準 tokio タスクから直接呼び出せない。チャネル経由の委譲が必須になるが、
   そのコストが io_uring のメリットを消す。

3. **LRU キャッシュが既に高速パスを提供している**: キャッシュヒット時は I/O 不要。
   キャッシュミス時のみ I/O が発生するが、そこで io_uring が遅い。

4. **io_uring が有効なシナリオはこのアーキテクチャに存在しない**:
   io_uring の真の利点は「多数の op を1 syscall でバッチ投入」だが、
   gRPC 1 RPC = 1 I/O op の構造では恩恵がない。
   高並列負荷下でのバッチ効果も、チャネルがシリアライズするため活きない。

### 維持するもの

この実装試行で副次的に追加した改善は残す：
- `write_block` に `sync_all()` を追加（書き込みの耐久性向上）
- `spawn_blocking` を使った非同期ラッパー（tokio スレッドのブロッキングを回避）

---

## 代替案

将来もし I/O がボトルネックになった場合に検討すべき施策：
- **LRU キャッシュ容量の拡大**: 最も効果的。現状のアーキテクチャで有効。
- **`tokio::fs`**: `spawn_blocking + std::fs` のラッパー。今の実装と同等。
- **io_uring SQPOLL モード + 固定バッファ**: 極めて複雑だが真のゼロコピー。
  1 RPC = 1 block という構造を根本から変える必要がある。
